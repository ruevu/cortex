### 2. Tool-input UX fixes at the MCP layer

**Status:** no spec yet — write one before executing.

**What:** Make the MCP server's `get_code_snippet` (and ideally `why_was_this_built`, `trace_path`) accept raw file paths and auto-resolve to canonical qns. Today they require the exact slash-replaced project-prefixed dotted form. The CLI's `resolve-input.ts` handles this for CLI users; this work makes the SAME logic available via the MCP protocol so agentic users (Claude Code over MCP) benefit too.

**Why:** The 2026-05-21 follow-up field report's "get_code_snippet returns No results on .vue paths" complaint is real for MCP callers. The CLI work alone doesn't fix it; the agent inside Claude Code is talking MCP, not running CLI commands.

**Shape of the fix:**
- Extract the heuristic from `src/cli/resolve-input.ts` into a shared module (e.g. `src/shared/resolve-input.ts`)
- Update `get_code_snippet` handler in [src/mcp-server/tools/code-tools.ts:271](../src/mcp-server/tools/code-tools.ts#L271) to call the resolver before the SQL lookup
- Same for `trace_path` and `why_was_this_built`
- Update the tool descriptions to mention "accepts file paths, qualified names, or bare symbol names"

**Estimated effort:** smaller than a full plan — likely 1 spec page + 4 tasks. Brainstorm if uncertain, otherwise write the plan directly.

## Reference: what the harness found

For anyone resuming, the harness's first run on anthill-cloud produced 9 surprises. After the investigation in this session, they reduce to:

**Cypher-engine artifacts (not real bugs in extraction):** `vue_function_has_high_degree`, `vue_function_node_count_nonzero`, `vue_file_is_module_node`, `composable_has_callers`, `sfc_qn_well_formed`. The data is in the graph; the field report's queries failed because they used `f.degree` as a property.

**Tool-input UX bugs (queue item 2 above):** `get_code_snippet_returns_vue_content`, `search_graph_finds_vue_component`. The tools work with canonical qns but reject raw file paths.

**Real data gaps (still need work):** HTTP_CALLS edges = 0 (highest single-fix value, indexer C work — separate from queue items 1-3), HANDLES edges = 0, `pinia_store_node_exists` = 0, `decision_node_count_nonzero` = 0.

The HTTP_CALLS extraction work is **not** in any of the three queued items above. It's indexer (C) work and the largest single feature on the table. Worth spec'ing as the fourth follow-up once 1-3 ship.

## Open notes / nice-to-haves spotted

- The harness's `imports_edge_count_grew` assertion is intentionally absent from the static catalog — it needs a baseline-relative dynamic predicate. ~30 lines to add to the assertion runner. Track as a small follow-up.
- `targets.json` for the harness uses `sha: "main"` as a placeholder. Pin real SHAs before running on `nuxt-ui` or `nuxthub-starter` for reproducibility.
- The harness's `maybeReindex` mtime check uses `.git/HEAD`, which doesn't update on `git pull`. For local-path targets, this can serve stale results. Document or fix.
- Anthill-cloud fixture's `vue_component_name: "app"` matches 450 nodes — replace with something more specific (e.g. `ArcaneChatMarkdown`) before relying on that assertion.
- `resolveCortexDbPath` returns `.cortex/db` but the actual file is `.cortex/graph.db` — internal contradiction in [src/db/resolve-path.ts](../src/db/resolve-path.ts). Same bug the 2026-05-20 field assessment flagged.
